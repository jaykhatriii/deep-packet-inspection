# DPI Engine — Node.js

A Deep Packet Inspection engine for Node.js. It reads a `.pcap` capture,
classifies each flow (via SNI / HTTP Host / DNS / port heuristics), applies
blocking rules, and writes the forwarded packets to an output `.pcap` along with
a summary report.

No external dependencies — pure Node.js (`>= 14`, tested on Node 26).

## Layout

```
js/
├── lib/
│   ├── types.js          # Data structures (FiveTuple, AppType, hashing)
│   ├── pcapReader.js      # PCAP file reading
│   ├── packetParser.js    # Ethernet / IPv4 / TCP / UDP parsing
│   └── sniExtractor.js    # TLS SNI / HTTP Host / DNS / QUIC inspection
├── dpiSimple.js           # Single-threaded engine (v1.0)
└── dpiMt.js               # Load-balanced pipeline engine (v2.0)
```

## Usage

```bash
# Single-threaded engine
node js/dpiSimple.js <input.pcap> <output.pcap> [options]

# Load-balanced pipeline engine (v2.0)
node js/dpiMt.js <input.pcap> <output.pcap> [options]
```

Options:

| Flag | Meaning |
|------|---------|
| `--block-ip <ip>` | Block all traffic from a source IP |
| `--block-app <app>` | Block an application (`YouTube`, `Facebook`, `TikTok`, …) |
| `--block-domain <dom>` | Block any SNI/Host containing this substring |
| `--lbs <n>` | Load-balancer count — `dpiMt.js` only (default 2) |
| `--fps <n>` | Fast-path processors per LB — `dpiMt.js` only (default 2) |

Example:

```bash
node js/dpiMt.js ../test_dpi.pcap out.pcap \
  --block-app YouTube --block-ip 192.168.1.50 --block-domain facebook
```

The bundled `../test_dpi.pcap` fixture works as-is (generated by
`../generate_test_pcap.py`).

## The two engines

- **`dpiSimple.js`** processes packets sequentially with a single flow table.
  Best for learning the pipeline and for small captures.

- **`dpiMt.js`** models a load-balanced, multi-stage pipeline:
  Reader → Load Balancers → Fast Paths → Output. It runs on a single thread —
  consistent hashing pins every packet of a flow to the same Fast Path, so each
  flow's classification, blocking decision and the per-stage counters are fully
  determined by the five-tuple hash, independent of timing. The "THREAD
  STATISTICS" section of the report shows how flows are distributed across the
  configured Load Balancers and Fast Paths.

Both engines produce the same set of forwarded packets and write them to the
output pcap in input order.

## Notes

- Only IPv4 + TCP/UDP over Ethernet is parsed; other packets are skipped.
- Byte order is detected from the pcap magic and used for both reading and
  writing, so little- and big-endian capture files round-trip correctly.
- The five-tuple hash is computed in 64-bit unsigned arithmetic (`BigInt`), so
  the load-balancing distribution is stable and deterministic.
- The application classifier uses simple substring matching, which has quirks:
  `www.netflix.com` and `www.microsoft.com` are labelled `Twitter/X` because
  they contain `x.com` and `t.co` respectively.
- Per-app counts and detected SNIs are kept in insertion-ordered `Map`s, so the
  report ordering is stable run to run.
